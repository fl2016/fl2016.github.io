POSIX共享内存

Posted on 2018-09-17

POSIX共享内存

创建共享内存对象

　　shm_open()函数创建或者打开一个新的共享内存对象或者打开一个既有对象。

#include <fcntl.h>
#include <sys/stat.h>
#include <sys/mman.h>
int shm_open(const char *name,int oflag,mode_t mode);

　　name参数标识出了待创建或待打开的共享内存对象，oflag参数是一个改变调用行为的位掩码，该参数表明了调用进程在共享内存上的访问模式，其取值为O_RDONLY或O_RDWR。
　　在一个新共享内存对象被创建时，其所有权和组所有权将根据调用shm_open()的进程的有效用户和组ID来确定，对象权限将根据mode参数中设置的位掩码来设定，mode参数的取值与文件权限上的取值一样，根据进程的umask取值，与open()不同的是，在调用shm_open()时总需要mode参数，在不创建新对象时需要将这个参数值指定为0.
　　shm_open()返回的文件描述符会设置close-on-exec标记，因此，当程序执行了一个exec()时文件描述符会自动关闭。

删除共享对象

　　　SUSv３要求POSIX对象至少具备内核持久性，即他们会持续存在直到被显示删除或者重启，使用shm_unlink()删除。

#incldue <sys/mman.h>
int shm_unlink(const char *name);

　　shm_unlink()函数会删除通过name指定的共享内存对象，删除一个共享内存对象不会影响现有的映射(他会保持有效直到相应的进程调用munmap()为止),但会阻止后续的shm_open()调用打开这个对象，一旦所有进程都解除映射这个对象，这个对象就会删除，其中的内容就会丢失。

文件加锁
  • flock()对整个文件加锁
  • fcntl()对一个文件区域加锁
    　　使用这两个函数的常规做法如下：
    　　　1.给文件加锁
    　　　2.执行文件I/O
    3.解锁文件以使得其他进程可以给文件加锁
混合使用加锁和stdio函数

　　由于stdio库会在用户空间中进行缓冲，一般情况下使用以下办法：
　　　1.使用read()和write()以及相关的系统调用取代stdio函数库来执行文件I/O
　　　2.在对文件加锁之后立即刷新stdio流，并且在释放锁之前立即再次刷新这个流。
　　　3.使用setbuf()或类似的函数来禁用stdio缓冲，可能会牺牲一些效率。 　　　

flock()给文件加锁
#incldue <sys/file.h>
int flock(int fd,int operation);

　　flock系统调用会在整个文件上放置一个锁，待加锁的文件是通过传入fd的一个打开着的文件描述符来指定的。
　　在默认情况下，如果一个进程已经持有了文件上一个不兼容的锁，那么flock()将会阻塞，如果需要防止出现这种情况，可以在operation参数中对这些参数取OR，这样即使另一个进程已经持有了文件上一个不兼容的锁，那么flock()就不会阻塞。
　　任意数量的进程可同时持有一个文件上的共享锁，但同一个时刻只有一个进程可以持有该文件上的互斥锁。
　　
　　无论一个进程在一个文件上的访问模式是什么，它都可以在该文件上放置一个共享锁或者互斥锁。
　　
　　通过再次调用flock()并在operation参数中指定恰当的值就可以将一个既有共享锁转换成互斥锁，将一个共享锁转换成互斥锁，在另一个进程持有该文件的共享锁时就会阻塞，除非同时指定了LOCK_NB标志。
　　锁的转换并不一定是原子的，在转换过程中，首先会 删除现有的锁，然后创建一个新的锁，在这两部之间另一个进程对一个不兼容锁的请求可能会得到满足，如果发生了这种情况，那么转换过程可能会阻塞，或者指定LOCK_NB标志时转换过程会失败并且进程会丢失原先的锁。 　　

锁继承和释放的语义

　　根据flock()调用 并将operation参数指定为LOCK_UN可以释放一个文件锁，此外，锁会在相应的文件描述符被关闭之后自动释放，通过flock()获取的文件锁是与打开的文件描述而不是文件描述符或文件(i-node)本身相关联的。当一个文件描述符被复制时，新文件描述符会引用同一个锁，如果通过一个特定的文件描述符获取一个锁并创建了该文件描述符的一个或者多个副本，那么如果不显式的执行一个解锁操作，只有所有的描述符副本都被关闭之后锁才会释放。
　　通过flock()创建的锁在exec()中会得到保留(除非在文件描述符上设置了close-on-exec标志并且该文件描述符是最后一个引用底层打开的问阿金描述的文件描述符)。
　　使用flock()只能添加劝告锁。 　　

使用fcntl()给文件加锁

　　可以在文件的任何一部分可以是一个字节，也可以是整个文件加锁，这种形式的文件加锁称为记录加锁。
　　fcntl()会用来锁住文件中与应用程序定义的记录边界对应的字节范围

flock结构
struct flock{
short l_type;//放置锁的类型
short l_whence;//指定了待加锁的范围，与l_start一起
off_t l_start;
off_t l_len;//包含了一个指定待加锁的字节数的整数
pid_t l_pid;
};

　　l_type字段的锁的类型总共有3中：
　　　　1.F_RDLCK 放置一把读锁
　　　　2.F_WRLCK 放置一把写锁
　　　　3.F_UNLCK 删除一把既有锁
　　l_start指定了文件的偏移量，l_whence 为SEEK_SET时，为文件的起始位置，为SEEK_CUR时，为当前文件的偏移量，SEEK_END时，为文件的结尾位置。
　　l_len指定为0具有特殊意义，即对”范围从l_start到l_whence确定的起始位置到文件的结尾位置的所有字节加锁，不管文件增长有多大”。
　　关于fcntl()创建的锁有以下几点：

  • 解锁一块文件区域总是会立即成功，即使当前不持有该区域上的一个锁，对这块区域解锁也不是一个错误。
  • 在任意一个时刻，一个进程只能持有一个文件特定区域上的一种锁，在之前已经锁住的区域上放置一把新锁不会发生任何事情(新锁与既有锁的类型是一样的)或原子的将既有锁转换成新模式，当将一个读锁转换成写锁时需要为调用返回一个错误(F_SETLK)或阻塞(F_SETLKW)做好准备。
  • 一个进程永远都无法将自己锁在一个文件区域中，即使通过多个引用同一个文件的文件描述符放置锁也是这样。
  • 在已经持有的锁中间放置一把模式不同的锁会产生3把锁：在新锁的两端会创建两个模式较之前模式更小一点的锁。
死锁

　　使用F_SETLKW时可能导致死锁，每个进程所请求的锁都被另一个进程持有的锁阻塞，如果内核不进行检查和抑制，那么会导致两个进程永远阻塞。内核会通过FSETLKW发起的每个新锁请求进行检查以判断是否会导致死锁，如果会导致死锁，那么内核就会选择其中一个阻塞的进程使其fcntl()调用解除阻塞并返回错误EDEADLK。 　　

锁继承和释放的语义

　　fcntl()记录锁和flock()创建的锁的继承和释放的语义是不同的：
　　　　1.由fork()创建的子进程不会继承记录锁，与flock()不同，使用flock()创建锁时，子进程会继承一个引用同一把锁的引用并且能够释放这把锁，从而导致父进程也失去这把锁。
　　　　2.记录锁在exec()中会得到保留，除非是设置了close-on-exec时会被exec()调用关闭
　　　　3.一个进程中的所有线程会共享同一组记录锁。
　　　　4.记录锁同时与一个进程和i-node节点关联，当一个进程终止之后，其所有记录锁都会被释放。这个和当一个进程关闭了一个文件描述符之后，进程持有的对应文件的所有锁都会被释放。

锁定饿死和排队加锁请求的优先级

　　在linux上，当多个进程放置一把锁时，排队的锁请求的顺序是不确定的，如果多个进程正在等待加锁，那么他们被满足的顺序取决于进程的调度。
　　写者并不会读者拥有更高的优先权。

强制加锁

　　以上所介绍的锁都是劝告式的锁，意味着一个进程可以自由忽略fcntl()和flock()的使用或者可以简单的执行I/O，内核不会阻止这种行为，使用劝告式锁时，设计者需要：
　　　　1.为文件设置合适的所有权或者组所有权，以及权限以防止非协作进程执行文件I/O。
　　　　2.通过在执行I/O之前获取恰当的锁来协作进程。
　　强制加锁必须在Linux上包含待加锁的文件的文件系统以及待加锁的文件上启用这一功能，通过在挂载文件系统时使用-o mand选项能够使该文件系统上启用强制式加锁

#mount -o mand /dev/sda10 /testfs
mount | grep sda10 //查看一个挂载文件系统是否启用了强制式加锁

　　文件上的强制式加锁是通过开启set-group-ID权限位和关闭group-execute权限来执行的

chmod g+s,g-s /testfs/file
强制加锁对文件I/O的影响

　　对一个文件启用强制式加锁时，那么执行文件传输的系统调用(read或write)碰到锁冲突时（在当前被读或写操作锁的区域上执行一个写入操作或当前被写锁住的区域上执行一个读操作）。如果是以阻塞模式打开了文件，那么系统调用就会阻塞，如果非阻塞模式(设置了O_NONBLOCK标记)，那么系统调用就会立即失败并返回EAGAIN错误。
　　如果存在进程持有了一个文件任意部分上的强制式读锁或写锁，那么就无法在该文件上创建一个共享内存映射，同样，如果一个文件参与了共享内存映射，那么就无法在该文件的任意部分防止一把强制锁。
